0. System Overview
CafeFlow SaaS is a multi-tenant, multi-branch café management platform built on Next.js 14, PostgreSQL (with PLpgSQL stored procedures), TypeScript, Firebase, and Tailwind CSS. It supports 8 distinct roles, full offline mode via a local engine, Ethiopian VAT compliance, and Telebirr/CBE Birr payment integration.

1.4 Subscription & Billing Management
Route: /saas-admin/billing

Subscription Model
• Trial Period: 7 days free (no payment required)
• Subscription Fee: 30,000 ETB per 6 months (bi-annual)
• Payment Method: Manual bank transfer to SaaS Owner account
• Receipt Upload: Cafe Owner uploads bank receipt via dashboard
• Approval: SaaS Owner reviews receipt and manually approves
• Next Gate: Subscription window reappears at T-7 days before expiry
• Grace Period: 3 days after expiry before full lockout

▸ Approval Workflow
Cafe Owner uploads receipt (image/PDF, max 5MB) via subscription modal
SaaS Owner receives in-app notification + email alert
SaaS Owner opens /saas-admin/billing/approvals
Views receipt image, uploaded timestamp, cafe name, amount claimed
Clicks "Verify & Approve" — enters verification note + approves
System activates subscription: start_date = today, end_date = today + 180 days
Cafe Owner dashboard unlocks instantly — subscription gate dismissed
Confirmation email sent to Cafe Owner with new expiry date

Rejection Flow: SaaS Owner clicks "Reject" → enters rejection reason → Cafe Owner notified with reason + option to re-upload.

9. Complete Order Lifecycle
This section maps the full journey of a single order through all roles from placement to payment.

Order State Machine
• DRAFT → Customer/Waiter building the order
• SUBMITTED → Sent to system, pending KDS routing
• IN_PREPARATION → Barista/Kitchen working on items
• PARTIALLY_READY → Some items done, others pending
• READY → All items prepared, awaiting delivery
• DELIVERED → All items confirmed delivered by Waiter
• BILL_REQUESTED → Waiter sent bill request to Cashier
• PAYMENT_PENDING → Cashier opened bill, awaiting payment
• COMPLETED → Paid, receipt issued, table freed
• VOIDED → Cancelled by Manager (pre-payment)
• REFUNDED → Refund processed post-payment

9.3 Offline Order Flow
Offline Mode Behavior
• Triggered automatically when internet connection drops
• All order data stored in IndexedDB via Dexie.js (local engine)
• Orders continue to flow between roles via local WebSocket (same WiFi)
• KDS screens still receive updates via local network
• Payments stored locally — processed online when connection restored
• Data sync: automatic on reconnect — conflict resolution via timestamp priority
• Offline badge shown in header for all users: "OFFLINE MODE (LOCAL ENGINE)"

18. Subscription Gate — Full Implementation Detail

18.1 State Machine
State
Condition & Access Level
TRIAL
Default on signup. trial_end = created_at + 7 days. Full access.
PENDING
Receipt uploaded, awaiting SaaS Owner approval. Read-only access.
ACTIVE
Approved. sub_end = approval_date + 180 days. Full access.
WARNING
sub_end - 7 days. Banner shown, modal shown on next login. Full access.
GRACE
sub_end passed. 3-day grace period. Degraded access (view only).
EXPIRED
Grace period over. Full lockout. Gate modal appears on all routes.
SUSPENDED
Manually suspended by SaaS Owner. Immediate lockout. No modal.

18.2 Middleware Implementation
Next.js Middleware Logic (middleware.ts)
• On every request to /owner/*, /manager/*, /waiter/*, /cashier/*, /barista/*, /kitchen/*, /store/*:
• 
• 1. Decode JWT → extract tenant_id
• 2. Call checkSubscriptionStatus(tenant_id) — PostgreSQL stored procedure
• 3. Returns: { state, trial_days_left, sub_days_left, grace_days_left }
• 4. If state === EXPIRED or SUSPENDED → redirect to /subscription/gate
• 5. If state === GRACE → allow request but inject X-Subscription-Grace header
• 6. If state === WARNING → allow request, inject X-Subscription-Warning header
• 7. Client layout reads headers → shows appropriate banner/modal
• 
• Subscription check is cached per tenant_id in Redis for 5 minutes to avoid DB hit on every request.
• Cache invalidated immediately on SaaS Owner approval action.
